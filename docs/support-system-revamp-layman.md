# Support System Revamp

## Purpose

This document explains, in simple non-technical language, how the new support system should work for customers, admins, founders, and brands.

The goal is to make support feel organized, fast, visible, and trustworthy from the moment an issue is raised until the final resolution is completed.

## Main Goal

We want a proper support system that:

- allows customers and brands to raise issues easily
- supports image attachments and proof uploads
- sends notifications to the right people immediately
- gives admin a real support workspace instead of a basic chat screen
- links support cases to orders, products, shipment tracking, and dispute handling
- allows brand action after admin approval
- supports replacement orders created from an existing order
- allows us to send goodwill discount coupons after resolution
- keeps the full case history visible from start to finish

## Who Will Use This System

There are four main groups who will use the support system:

- Customers
- Admin / Support team
- Brand admins
- Founders / internal escalation team

Each group needs a different view and different notifications.

## New Support Structure

The support system should be split into clear areas instead of mixing everything in one place.

### Customer Support

This is for users who face issues such as:

- order problems
- damaged item
- wrong item
- missing item
- delivery issue
- refund issue
- payment issue
- account issue

### Brand Support

This is for brands who need help with:

- payouts
- catalog or product issues
- account issues
- operations
- platform-related concerns

### Admin Support Console

This should be the main internal support workspace where the team manages customer and brand issues separately.

### Brand Dispute / Action Center

This should be a dedicated brand-side workspace where brands can see approved disputes and act on them.

## Customer Support Journey

When a customer faces a problem, the flow should work like this:

1. The customer opens the Help & Support area.
2. The customer selects the type of issue.
3. If the issue is order-related, the customer selects the exact order.
4. The system automatically shows relevant order details.
5. The customer explains the issue in a chat-style support form.
6. The customer uploads proof such as pictures or screenshots.
7. A support ticket is created.
8. The customer sees the ticket ID, current status, and next step.
9. Notifications are sent to the support team immediately.
10. The customer receives confirmation in the app and by email.
11. The customer can return later and continue the same conversation.

## What the Customer Should See in a Ticket

The ticket page should look like a proper case file plus chat. It should show:

- issue title
- ticket status
- priority
- linked order
- shipment status
- product details
- uploaded images
- conversation history
- timeline of updates
- final resolution summary when the case is completed

Suggested statuses:

- Open
- Under Review
- Waiting for Customer
- Waiting for Brand
- Approved
- Rejected
- Resolved
- Closed

## Attachment System

The new support system should allow proper proof uploads.

Customers should be able to attach:

- item photos
- packaging photos
- screenshots
- payment proof
- delivery proof
- invoice screenshots

Admins should be able to:

- view attachments inside the case
- open and review proof easily
- use attachments while making a decision

Later, the same attachment capability can also be extended to brands and admins where needed.

## Notifications Flow

Notifications are a key part of making support work well.

### When a customer raises a support issue

- Admin/support team gets an in-app notification
- Admin/support team gets an email
- Founders or escalation owners can also be notified for important cases

### When admin replies

- Customer gets an in-app notification
- Customer gets an email

### When the ticket status changes

- Customer gets an in-app notification
- Customer gets an email

### When a dispute is approved for brand action

- Brand admin gets an in-app notification
- Brand admin gets an email

### When brand completes its action

- Admin gets notified
- Customer can also be informed if required

### When the case is resolved

- Customer gets a final update
- Customer may also receive an apology coupon notification if one is sent

## Admin Support Console Revamp

The current admin support screen should be replaced with a more complete support workspace.

Instead of one mixed page, admin should have separate menus such as:

- User Support
- Brand Support
- Disputes
- Escalations

Inside the user support area, admin should have tabs like:

- All
- Open
- Under Review
- Waiting for Customer
- Waiting for Brand
- Approved
- Resolved
- Escalated

Brand support should have its own similar structure.

Every ticket row should clearly show:

- ticket ID
- issue type
- customer or brand name
- linked order if any
- ticket age
- priority
- unread count
- assigned owner if any
- current status

## Admin Ticket Detail Page

The admin view of a ticket should be much richer than basic chat.

It should include:

- full conversation
- customer or brand details
- linked order details
- shipment tracking details
- product details
- uploaded proof images
- internal notes
- decision buttons
- resolution history
- brand action status when relevant

Admin actions should include:

- reply to customer
- request more proof
- set priority
- assign the case
- escalate the issue
- approve dispute
- reject dispute
- resolve case
- notify brand
- send goodwill coupon
- add internal notes

## Linking Order and Tracking Inside Support

For order-related cases, admin should not need to leave the support page to investigate.

Inside the case, support should show:

- order summary
- item details
- quantity
- price
- shipping partner
- tracking number
- tracking timeline
- delivery status
- payment status
- refund status if relevant

This makes support faster and reduces confusion.

## Dispute Flow With Brand Involvement

If a customer issue becomes a dispute case, the flow should work like this:

1. Customer raises an issue and uploads proof.
2. Admin reviews the issue, order details, and attachments.
3. Admin decides whether the dispute is valid.
4. If valid, the case moves to an approved-for-brand-action stage.
5. Brand admin receives notification.
6. Brand opens the dispute from a dedicated dispute/action area.
7. Brand sees the original order, proof, admin decision, and required next step.
8. Brand takes the required action.
9. If a replacement is needed, the brand creates a replacement order from the original order.
10. Admin and customer can track what happens next.

## Brand Admin Side Requirements

Brand admins should have a clean workspace for dispute-related cases.

Suggested menus:

- Support
- Disputes
- Replacement Orders
- Resolution History

For each approved dispute, the brand should see:

- original order
- issue summary
- proof images
- admin decision
- required next action
- action deadline if needed
- replacement order action button
- dispute history

The brand should only take dispute action after admin approval.

## Replacement Order Flow

When a replacement needs to be created from a dispute:

- the new order should be created from the original order
- customer details should already be filled in
- shipping address should already be filled in
- product details should already be filled in
- the new order should stay linked to the support case
- the new order should be marked clearly as a dispute-related replacement
- admin and customer should be able to track it

This avoids mistakes and makes the recovery process faster.

## Resolution Flow

A support case should not simply end with the word "resolved." The system should clearly record what happened.

The final outcome should mention:

- what the issue was
- what decision was taken
- whether refund, replacement, apology, or explanation was given
- whether brand action was completed
- whether the customer was informed

Suggested resolution types:

- Resolved with explanation
- Resolved with refund
- Resolved with replacement
- Resolved with coupon
- Rejected with reason
- Closed due to inactivity

## Personalized Apology Coupon Flow

After a case is resolved, admin should have the option to send a personalized goodwill coupon.

This can be used for:

- customer inconvenience
- delayed delivery
- damaged item apology
- service recovery
- retaining a frustrated customer

Suggested flow:

1. Admin chooses to send an apology coupon.
2. Admin selects the type of coupon.
3. Admin adds a short message.
4. System sends the coupon by in-app notification and email.
5. The case history records that the coupon was sent.

## Internal Notes

Admins should be able to write internal notes that customers and brands cannot see.

These notes can help the team coordinate, for example:

- customer seems genuine
- waiting for courier verification
- high-priority case
- approve replacement if proof matches
- founder asked to prioritize

## Priority and Escalation

The system should support different levels of urgency:

- Low
- Normal
- High
- Critical

Critical cases may include:

- fraud risk
- legal concerns
- social media escalation
- VIP customer issue
- repeated order failures
- severe brand dispute

Critical cases should notify founders or internal escalation owners immediately.

## Reporting and Visibility

Once the main support flow is in place, the system should also help the team track performance through reports such as:

- total open tickets
- average first response time
- average resolution time
- disputes by brand
- most common issue categories
- unresolved high-priority cases
- replacement case count
- coupon recovery usage

### Phase 1

- new support UI
- separate admin menus for user and brand support
- attachment upload support
- better ticket statuses
- app and email notifications

### Phase 2

- order and shipment details inside tickets
- internal notes
- improved admin workflow
- brand dispute menu

### Phase 3

- admin-approved dispute to brand action flow
- replacement order creation from existing order details

### Phase 4

- personalized apology coupon flow

## Recommended Extra Improvements

In addition to the main flow above, the following improvements are strongly recommended:

- SLA timers so support knows how long a case has been waiting
- unread badges everywhere they matter
- support case assignment
- canned replies for common issues
- internal tags such as refund, replacement, courier, urgent
- action history / audit trail
- customer satisfaction rating after resolution
- ticket reopen option for a limited period

## Final Summary

In simple terms, the new support system should work like this:

A customer reports a problem, uploads proof, and gets a proper support case. Admin immediately sees the issue with order and tracking details, reviews the case, chats with the customer, and decides what should happen next. If the brand needs to act, admin approves the dispute and notifies the brand. The brand then sees the approved case and can create a replacement order directly from the original order details. Once the issue is solved, the customer is updated and can also receive a personalized apology coupon if needed. Every major step should trigger the right notifications so the case never gets lost.
